# Benchmark methodology

The project's thesis — *native arm64 biocontainers are meaningfully faster (and,
on Graviton, cheaper) than amd64-under-emulation* — is only worth telling if it
is **measured, reproducible, and resistant to "you cherry-picked that" attacks.**
This document is the protocol. It is written before any numbers exist so the
method can't be reverse-engineered from a desired conclusion.

> **Status:** methodology + harness only. No results collected yet. Results, when
> produced, land in `results/` with the full environment captured alongside.

## The single claim under test

> For the same tool, same version, and same input, a **native arm64** container
> runs faster than the **amd64 container under QEMU emulation**, on the same
> machine. On Graviton, it is additionally cheaper per workload.

The *only* variable is the container architecture / execution mode. Everything
else — tool version, input data, thread count, machine — is held fixed within a
given comparison.

## Two legs

| Leg | Machine | Compares | Cost | Status |
|-----|---------|----------|------|--------|
| **Mac** | Apple Silicon (this repo's dev box: M4 Pro, 12c, 48 GB) | amd64-via-QEMU vs native arm64 | free | harness ready, not run |
| **Graviton** | AWS EC2 (e.g. c7g vs c7i), via spore.host / nf-spawn | native arm64 vs amd64, *and* $/workload | **real money** | plan only — requires explicit sign-off, see [graviton-plan.md](graviton-plan.md) |

The Mac leg proves *speed*. The Graviton leg proves *speed + cost* and ties to
the spore.host ephemeral-compute story (cheaper than people assume).

## Tools under test

Chosen to span the emulation-sensitivity spectrum, so the result is honest about
*where* the benefit is large and where it's modest:

| Tool | Type | Expected emulation tax | Why included |
|------|------|------------------------|--------------|
| `bwa mem` | compiled C, SIMD | high (~2–4×) | the classic aligner; worst-case native hotloop |
| `minimap2` | compiled C, SIMD | high (~2–4×) | modern aligner; SIMD-heavy, QEMU's weak spot |
| `samtools sort/view` | compiled C | medium–high | ubiquitous in every pipeline; self-contained |
| `seqkit` | Go binary | moderate | fast, trivial self-contained workload; lower-bound case |

Reporting a *moderate* case (seqkit) next to *high* cases (bwa/minimap2) is
deliberate — it shows we're not hiding the tools where the benefit is smaller.

## Controls (what makes it honest)

1. **Identical tool version** in both images, pinned exactly (`bwa=0.7.18`, etc.).
   The native arm64 image is built the *project's own way* (`conda install
   <pkg>=<ver>` on arm64), so the benchmark doubles as a proof of the build
   method.
2. **Identical input**, generated by a seeded synthesizer (`gen_data.sh`) so
   anyone can reproduce the exact bytes — no private data, no network fetch.
3. **Pinned thread count** (`--threads`/`-t` fixed, default 4) so we measure CPU
   work, not scheduler variance. Run once at 1 thread and once at N to show
   whether the tax scales.
4. **Warm cache**: one untimed warm-up run, then `N=10` timed runs. Report
   **median** and **min–max**, never a single number or a mean (means hide
   outliers).
5. **Wall-clock + max-RSS** captured per run (`/usr/bin/time -l` on macOS).
   Memory matters: emulation can inflate RSS, and Graviton memory-bandwidth is
   part of the cost story.
6. **Same base layers** where possible, so we compare the tool, not the OS image.
7. **Environment captured** into every result file: chip, core count, RAM, OS
   build, Docker version, QEMU/binfmt version, image digests (by sha256, not
   tag). A result with no captured environment is discarded.

## Execution modes on the Mac leg

Three rows per tool, all on the same M-series machine:

- **`amd64-emulated`** — `docker run --platform linux/amd64 quay.io/biocontainers/<tool>` (QEMU). The status quo most Mac users are silently on.
- **`arm64-native`** — `docker run --platform linux/arm64 quay.io/playground-logic/<tool>` (this project's image).
- **`arm64-bare`** *(optional baseline)* — the conda-installed binary run directly on the host, no container — to isolate container overhead from architecture.

## What we will NOT claim

- We will **not** extrapolate one tool's speedup to "biocontainers are Nx
  faster." Each tool reports its own number; the headline is a range.
- We will **not** compare across different machines and call it an arm-vs-amd
  result (an M4 beating an old Xeon proves nothing about architecture).
- We will **not** report cost savings from the Mac leg — cost only comes from the
  Graviton leg against real EC2 on-demand/Spot prices.
- We will flag any tool where native arm64 is *not* faster, if that happens.
  Negative results stay in the table.

## Repro

```
benchmark/
├── METHODOLOGY.md      # this file
├── gen_data.sh         # seeded, deterministic test inputs (no network)
├── run_mac.sh          # Mac leg: emulated vs native, N runs, captures env
├── graviton-plan.md    # Graviton leg design — NOT run without sign-off
└── results/            # output JSON + captured environment (git-tracked)
```

Anyone can re-run `gen_data.sh && run_mac.sh` on their own Apple Silicon machine
and get comparable numbers. That reproducibility *is* the trust argument — it's
the same principle as D6 (trust the verifiable process, not the publisher).
